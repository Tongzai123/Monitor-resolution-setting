# Twinkle Tray 分辨率切换初版方案

## 目标

在 Twinkle Tray 现有右下角托盘体验中，增加显示器分辨率和刷新率的快速切换能力，让 4K 显示器用户可以在办公、游戏、投屏、兼容性场景之间快速切换显示模式。

初版目标不是替代 Windows 设置，而是提供一个轻量、稳定、可回滚的快速入口。

## 可行性结论

方案可行，且适合基于 Twinkle Tray 改造。

Twinkle Tray 当前产品定位是 Windows 托盘显示器控制工具，已经围绕显示器枚举、托盘面板、设置页、快捷键和显示器状态管理建立了基础能力。分辨率切换与亮度控制属于同一类“显示器快速控制”场景，放在同一个托盘面板中符合用户预期。

技术上，亮度控制主要依赖 DDC/CI 等显示器控制能力；分辨率切换不应走 DDC/CI，而应走 Windows 显示配置 API。初版可以使用 `EnumDisplaySettings` / `EnumDisplaySettingsEx` 枚举显示模式，使用 `ChangeDisplaySettingsEx` 应用指定显示器的分辨率和刷新率。后续如果要支持显示器排列、主屏切换、复制/扩展模式等复杂能力，再评估使用 `QueryDisplayConfig` / `SetDisplayConfig`。

## 初版范围

### 必做功能

- 枚举每台显示器当前分辨率和刷新率。
- 枚举每台显示器支持的分辨率、刷新率组合。
- 在托盘主面板中显示当前分辨率，例如 `3840×2160 · 60Hz`。
- 支持对单台显示器切换分辨率和刷新率。
- 切换后显示倒计时确认，用户未确认则自动回滚。
- 支持收藏常用分辨率。
- 设置页提供总开关，允许用户关闭分辨率控制入口。

### 暂不纳入初版

- DPI 缩放比例切换。
- HDR 开关。
- 色深、颜色格式、动态范围切换。
- 显示器排列、主屏、副屏、复制、扩展模式管理。
- 按应用自动切换分辨率。
- 游戏启动检测和自动场景切换。
- 云同步或跨设备同步。

## 技术方案

### 显示器识别

初版需要建立 Twinkle Tray 内部显示器对象与 Windows 显示设备之间的映射关系。

建议优先使用以下字段组合匹配：

- Windows display device name，例如 `\\.\DISPLAY1`。
- 设备路径或适配器路径。
- EDID 信息中的厂商、型号、序列号。
- Twinkle Tray 已有的显示器 ID。

匹配策略应允许降级：当 EDID 或序列号不可用时，仍可通过 Windows display name 和当前连接顺序提供基础功能，但 UI 中要避免承诺“永久绑定”。

### 模式枚举

初版建议枚举以下信息：

- 宽度。
- 高度。
- 刷新率。
- 色深。
- 当前模式标记。
- 是否推荐模式。

模式列表需要去重，避免同一分辨率和刷新率因为底层字段差异重复出现。

默认展示策略：

- 优先显示当前模式。
- 优先显示常见比例和常见刷新率。
- 默认隐藏低于 `60Hz` 的模式，设置中允许用户打开。
- 默认隐藏明显不适合当前显示器比例的模式，设置中允许显示全部。

### 应用模式

初版建议使用 `ChangeDisplaySettingsEx` 应用指定显示器模式。

关键要求：

- 应用前记录原始模式。
- 应用后启动确认倒计时。
- 用户点击“保留更改”后才写入收藏或场景。
- 倒计时结束、应用失败或窗口失焦异常时，自动尝试还原原始模式。
- 应用失败时显示清晰错误，不吞掉异常。

### 回滚机制

分辨率切换属于高风险交互，因为错误模式可能导致黑屏、画面超出范围或远程桌面断连。

建议初版固定使用 15 秒倒计时：

- 切换成功后显示确认条。
- 用户点击“保留更改”则结束倒计时。
- 用户点击“还原”则立即恢复原始模式。
- 倒计时结束未确认则自动还原。
- 如果还原失败，提示用户使用 Windows 显示设置手动恢复。

## UI 方案

### 托盘主面板

保持 Twinkle Tray 现有显示器卡片结构，不把分辨率功能做成独立大面板。

建议每台显示器显示如下：

```text
LG 27UP850                         设置
亮度  ━━━━━━━━━━━━━━━  68%

分辨率   3840×2160 · 60Hz       展开
```

分辨率行默认只显示当前值，减少对原有亮度操作的干扰。

### 分辨率下拉面板

点击当前分辨率后，展开轻量选择面板：

```text
常用
✓ 3840×2160 · 60Hz
  2560×1440 · 60Hz
  1920×1080 · 60Hz

全部模式
  3840×2160 · 30Hz
  2560×1440 · 75Hz
  1920×1080 · 120Hz
```

交互建议：

- 当前模式使用选中态。
- 收藏模式放在顶部。
- 不推荐把所有模式一次性铺满，列表过长时使用滚动区域。
- 刷新率必须和分辨率放在同一行，避免用户误以为只改分辨率。
- 对低刷新率或异常比例模式可使用较弱视觉样式。

### 切换确认条

切换成功后，在托盘面板底部或对应显示器卡片下方显示确认条：

```text
已切换到 2560×1440 · 60Hz
保留更改    还原    14 秒后自动还原
```

确认条要求：

- 视觉上足够明显。
- 不遮挡亮度滑块。
- 倒计时实时更新。
- 面板关闭时仍继续倒计时，并按规则自动回滚。

### 设置页

设置页增加“分辨率”区域：

```text
分辨率
[开关] 在主面板显示分辨率控制
[开关] 显示刷新率
[开关] 隐藏低于 60Hz 的模式
[开关] 只显示收藏模式
[按钮] 管理收藏分辨率
```

高级选项可后置：

```text
高级
[开关] 显示所有 Windows 报告的显示模式
[输入] 自动还原倒计时：15 秒
[按钮] 重新扫描显示器模式
```

### 托盘右键菜单

右键菜单只放常用模式，避免菜单过长：

```text
LG 27UP850
  3840×2160 · 60Hz
  2560×1440 · 60Hz
  1920×1080 · 60Hz
```

如果用户未配置收藏模式，右键菜单可以只显示当前模式和“更多分辨率...”入口。

## 场景预设

初版可以先不实现完整场景系统，但数据结构应预留空间。

建议后续支持：

```text
办公：3840×2160 · 60Hz，亮度 45%
游戏：2560×1440 · 144Hz，亮度 80%
投屏：1920×1080 · 60Hz，亮度 60%
```

场景预设的价值高于单纯分辨率列表，因为用户实际需要的是“切换到某种使用状态”，而不是记住每个显示模式参数。

## 数据设计建议

初版配置可以按显示器保存：

```json
{
  "resolutionControlsEnabled": true,
  "hideLowRefreshRates": true,
  "showOnlyFavorites": false,
  "revertTimeoutSeconds": 15,
  "displayModes": {
    "display-id": {
      "favorites": [
        {
          "width": 3840,
          "height": 2160,
          "refreshRate": 60
        },
        {
          "width": 2560,
          "height": 1440,
          "refreshRate": 144
        }
      ]
    }
  }
}
```

显示器 ID 不应只依赖连接顺序，否则多显示器插拔后容易错配。应尽量使用 EDID、设备路径和 Windows display name 的组合。

## 风险与应对

### 黑屏或无信号

风险：用户选择了显示器无法稳定输出的模式。

应对：

- 必须实现倒计时自动回滚。
- 默认只显示 Windows 报告的可用模式。
- 高风险模式放入“显示全部模式”之后。

### 多显示器错配

风险：对 A 显示器的设置应用到了 B 显示器。

应对：

- 先建立可靠的显示器映射。
- UI 中显示显示器名称和编号。
- 应用前后校验目标显示器当前模式是否变化。

### 刷新率误选

风险：用户只想降低分辨率，却误选了低刷新率。

应对：

- 分辨率和刷新率必须作为一个完整模式显示。
- 默认隐藏低于 `60Hz` 的模式。
- 收藏模式优先。

### 远程桌面场景

风险：远程桌面、虚拟显示器或投屏环境下，显示模式枚举和应用行为可能异常。

应对：

- 检测到虚拟显示器时降低功能承诺。
- 应用失败时保留错误信息。
- 不对未知设备自动应用场景。

## 实施里程碑

### 第一阶段：原型验证

- 读取当前显示器分辨率和刷新率。
- 枚举支持的显示模式。
- 在开发调试界面展示模式列表。
- 对单台显示器执行切换。
- 实现自动回滚。

验收标准：

- 能正确识别至少一台 4K 显示器。
- 能在 `3840×2160`、`2560×1440`、`1920×1080` 等模式间切换。
- 未确认时能自动恢复原始模式。

### 第二阶段：接入托盘面板

- 在显示器卡片中显示当前分辨率。
- 增加分辨率下拉选择。
- 增加切换确认条。
- 保持原有亮度功能不受影响。

验收标准：

- 不改变用户原本调亮度的主流程。
- 分辨率切换入口清晰但不喧宾夺主。
- 切换失败时有明确提示。

### 第三阶段：设置和收藏

- 增加分辨率功能总开关。
- 增加收藏分辨率。
- 增加隐藏低刷新率选项。
- 增加重新扫描显示模式按钮。

验收标准：

- 用户可以只显示常用模式。
- 多显示器收藏互不干扰。
- 重启应用后设置保持。

### 第四阶段：快捷操作

- 托盘右键菜单显示常用分辨率。
- 支持快捷键切换收藏模式。
- 预留场景预设结构。

验收标准：

- 用户可以在不打开完整面板的情况下切换常用模式。
- 快捷键不会与现有亮度快捷键冲突。

## 推荐开发顺序

1. 先完成 Windows 显示模式枚举和切换的最小闭环。
2. 再把当前分辨率显示到 Twinkle Tray 面板。
3. 然后实现切换确认和自动回滚。
4. 最后再做收藏、设置项、右键菜单和快捷键。

这个顺序可以优先验证最核心风险：Windows API 是否能稳定识别并切换目标显示器。如果底层能力不稳定，UI 做得再完整也没有意义。

## 初版判断

最值得做的版本是“轻量分辨率控制”，不是“完整显示设置中心”。

建议初版只解决一个核心问题：用户在 4K、2K、1080p 以及不同刷新率之间快速、安全地切换。只要这个闭环稳定，后续再扩展场景预设、快捷键和自动化会更稳。
